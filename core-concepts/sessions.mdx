---
title: "Sessions"
description: "The lifecycle of a verification — from create to a terminal result."
---

A **session** is one user-facing verification flow containing one or more
[primitive](/core-concepts/primitives) steps. You create it server-to-server; the
user completes it on UIP's hosted page; you receive results by webhook.

## Lifecycle

<Steps>
  <Step title="created">
    `POST /v1/sessions` returns the session and its hosted `url`. Nothing is charged.
  </Step>
  <Step title="in_progress">
    The user opens the hosted page and begins; each step's wallet response is
    verified and emits `step.completed`.
  </Step>
  <Step title="awaiting_decision (optional)">
    If a step is a [gate](/core-concepts/primitives#gates-pause-for-a-decision), the
    session pauses here until you `continue` or `stop` it.
  </Step>
  <Step title="completed / stopped / failed / expired">
    A terminal state. The matching `session.*` webhook fires. Original files that
    participated in a completed `sign` step remain immutable evidence; only explicit
    transient uploads from uncompleted sign steps become cleanup-eligible.
  </Step>
</Steps>

| Status | Meaning |
| --- | --- |
| `created` | Made, not yet opened by the user |
| `in_progress` | User is working through the steps |
| `awaiting_decision` | Paused at a gate, awaiting your `continue`/`stop` |
| `completed` | All steps verified |
| `stopped` | You ended the active session before completion |
| `failed` | A step failed verification, funds ran out, or an internal failure ended it |
| `expired` | The session or a step window elapsed |

## The hosted page & capability URLs

The session `url` (`https://uip.digital/s/{id}`) is a **capability URL** — holding
the link is what lets a user open the session. Treat it like a secret: deliver it
over a channel only the intended user controls (your authenticated app, their phone).

<Warning>
  **First-open binding.** The first browser to open a session is cryptographically
  bound to it; only that browser can submit a wallet response. A second device that
  later opens the same link is refused. This stops a leaked link from being completed
  by anyone but the user who started it — so don't "preview" the URL server-side
  before handing it over.
</Warning>

## Reading a session

[`GET /v1/sessions/{id}`](/api-reference/sessions/get), authenticated with your API
key, returns status plus results already produced: requested `identify` claims,
`age_verify`'s boolean and threshold, and signing hashes/signatures. The unauthenticated
hosted page sees only its status projection and cannot read these results. Webhooks
remain the primary push channel; use GET to reconcile a missed event, not as a polling
loop.

## Create options

Beyond `steps`, `POST /v1/sessions` accepts:

| Field | Type | Purpose |
| --- | --- | --- |
| `client_reference_id` | string (≤200 chars) | Your own id, echoed on every webhook and stamped into the audit |
| `metadata` | object | Up to 50 string-valued keys (key ≤40 chars, value ≤500); operational only, never shown to the user |
| `return_url` | https URL | Renders a "Return to {host}" button on the hosted page after completion (never an auto-redirect) |
| `expires_in` | seconds | Override the session TTL (60s–24h) |

Plus an optional **`Idempotency-Key`** header so a retried create returns the
original session instead of a duplicate. See
[Create a session](/api-reference/sessions/create) for the full reference.

Keep `client_reference_id` non-PII because it is cleartext in the immutable audit.
Treat `metadata` as retained operational data, not a secret store.

## Expiry

Sessions have an overall TTL (1 hour by default, or a 60-second–24-hour
`expires_in` override), and each step has its own shorter window once started.
Expiry is enforced on the next state access, persisted, and enqueued as one logical
`session.expired` event (transport retries remain at-least-once).

## Result retention

Completed results and their audit records are legally significant service records.
They are immutable and are not customer-deletable. For a completed `sign`, this also
includes the exact original document bytes UIP hashed and presented to the signer.
Stopping or deleting a customer account does not erase them. Any account/data request
is reviewed for disposable data and legal holds; cleanup jobs may remove only data
explicitly classified as transient, such as an upload that never joined a completed
sign result.
