UIP uses a simple two-path UX based on the user's device. Your application detects the device type and shows the appropriate interface.

<CardGroup cols={2}>
  <Card title="Desktop flow" icon="desktop">
    Inline the raw SVG markup returned by the API. The user scans it with their UIP app.
  </Card>

  <Card title="Mobile flow" icon="mobile-screen">
    Show an "Open UIP" button that deep-links into the UIP app via the `uip://` URL scheme. No QR-code scan needed.
  </Card>
</CardGroup>

## Implementation flow

<Steps>
  <Step title="Create a session">
    Call `POST /v1/identify` or `POST /v1/sign` with your intent. The API returns `session_id`, `qr_code`, and `expires_at`.

    ```javascript
    const session = await fetch('https://api.uip.digital/v1/identify', {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${API_KEY}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        intent: 'Sign in to My App',
        requested_data: ['first_name', 'last_name']
      })
    }).then(r => r.json());
    // session = { session_id, qr_code, expires_at }
    // qr_code is a raw SVG string, NOT a base64 PNG data URI
    ```
  </Step>

  <Step title="Detect device type">
    ```javascript
    const isMobile = /iPhone|iPad|Android/i.test(navigator.userAgent);
    ```
  </Step>

  <Step title="Display the QR (desktop) or deep link (mobile)">
    **Desktop:** the `qr_code` field is **raw SVG markup** — inline it directly into the page. Do not set it as the `src` of an `<img>` tag; that won't work for SVG markup.

    **Mobile:** build a `uip://` deep link from the session details. Tapping it opens the UIP app directly.

    ```html
    <div id="uip-desktop"></div>

    <div id="uip-mobile" style="display: none;">
      <a id="uip-link" class="button">Open UIP</a>
    </div>

    <script>
      const isMobile = /iPhone|iPad|Android/i.test(navigator.userAgent);
      const expUnix = Math.floor(new Date(session.expires_at).getTime() / 1000);

      if (isMobile) {
        document.getElementById('uip-desktop').style.display = 'none';
        document.getElementById('uip-mobile').style.display = 'block';
        // Identify sessions: uip://identify?...
        // Sign sessions:     uip://sign?...
        document.getElementById('uip-link').href =
          `uip://identify?sid=${session.session_id}&exp=${expUnix}`;
      } else {
        // Inline the raw SVG markup. Frameworks: dangerouslySetInnerHTML in
        // React, {@html} in Svelte, v-html in Vue, |raw in Twig, etc.
        document.getElementById('uip-desktop').innerHTML = session.qr_code;
      }
    </script>
    ```

    Framework examples:

    ```jsx React
    <div dangerouslySetInnerHTML={{ __html: session.qr_code }} />
    ```

    ```svelte Svelte
    <div>{@html session.qr_code}</div>
    ```

    ```vue Vue
    <div v-html="session.qr_code"></div>
    ```

    <Warning>
    The `qr_code` value comes from a trusted source (UIP) and is generated server-side. You can inline it via `innerHTML` / `dangerouslySetInnerHTML` safely. If you have an XSS-strict CSP, ensure inline SVG is allowed.
    </Warning>
  </Step>

  <Step title="Poll for the result">
    Poll `GET /v1/identify/:id` (or `GET /v1/sign/:id`) every 2–3 seconds.

    - Identify terminal status: `"approved"` (or `"expired"`)
    - Sign terminal status: `"signed"` (or `"expired"`)

    The polling response carries the signer's identity snapshot and an `audit_id`. The full audit (AdES proof, attachments via signed URLs) is then available via [`GET /v1/audit/:id`](/api-reference/endpoint/audit).

    ```javascript
    async function pollIdentify(sessionId, apiKey) {
      while (true) {
        const res = await fetch(
          `https://api.uip.digital/v1/identify/${sessionId}`,
          { headers: { 'Authorization': `Bearer ${apiKey}` } }
        );
        const data = await res.json();

        if (data.status === 'approved') return data;
        if (data.status === 'expired') throw new Error('Session expired');

        await new Promise(r => setTimeout(r, 2500));
      }
    }
    ```
  </Step>

  <Step title="Use the signer snapshot or audit_id">
    The polling response gives you `target_uip_id` and `signer_snapshot` for immediate UI feedback. Save the `audit_id` if you need to retrieve the full audit (AdES proof, attachments) later via the [Audit API](/api-reference/endpoint/audit).
  </Step>
</Steps>

<Info>
**No webhooks needed for identify or sign.** Polling is the primary pattern. Webhooks are only used for the Message API.
</Info>

## QR-code payload format

The QR encodes a BankID-style string, not a URL:

```
uip|<session_id>|<unix_seconds_until_expiry>
```

The UIP app parses this format directly. There is no `https://` URL the user could open in a browser — scanning the QR with anything other than the UIP app does nothing meaningful.

## Deep-link URL scheme

For same-device flows where the user is already on a mobile browser, link to the `uip://` scheme rather than showing a QR they'd have to scan with the device they're already on:

```
uip://identify?sid={session_id}&exp={unix_seconds}
uip://sign?sid={session_id}&exp={unix_seconds}
```

Tapping the link opens the UIP app directly. iOS requires a `CFBundleURLTypes` entry for `uip`; Android requires an `<intent-filter android:scheme="uip">` on the receiving activity. If the user doesn't have the UIP app installed, the OS falls back to whatever the browser shows for unhandled custom schemes — typically a "no app to open this" dialog.

## QR-code display best practices

<AccordionGroup>
  <Accordion title="Size and positioning" icon="maximize">
    - **Minimum render size:** 200×200 px for reliable scanning from a phone
    - **Recommended:** 256–320 px square
    - **Quiet zone:** the SVG already includes its own quiet zone (margin). Don't override it with negative margins.
    - **Don't transform the SVG colors.** The UIP-issued QR is high-contrast by design; recoloring can break scannability.
  </Accordion>

  <Accordion title="Visual feedback" icon="spinner">
    - Show a loading state while creating the session
    - Display clear instructions: "Open the UIP app and scan"
    - Show a countdown to `expires_at` — sessions expire in ~5 minutes
    - When status flips to `approved` / `signed`, transition cleanly to your post-scan UI
  </Accordion>

  <Accordion title="Error states" icon="triangle-exclamation">
    - Handle `expired` status: surface a "Generate a new code" button
    - Handle network errors during polling: continue polling with backoff, surface "having trouble reaching UIP" after a few failures
    - Don't auto-retry past `expires_at` — generate a new session instead
  </Accordion>

  <Accordion title="Accessibility" icon="universal-access">
    - The inlined SVG has no built-in accessible label. Wrap it in a `<div>` with `role="img"` and an `aria-label` describing what to do, or surface text instructions alongside it.
    - Ensure sufficient contrast in the surrounding UI; the QR itself is already black-on-white.
  </Accordion>
</AccordionGroup>

## Session management

<CardGroup cols={2}>
  <Card title="Session expiration" icon="clock">
    Sessions expire after ~5 minutes (controlled by `SESSION_EXPIRY_SECONDS`, default 310s). Show the user a countdown.
  </Card>

  <Card title="One-time use" icon="shield-check">
    Each session is single-use. Once the user signs, the session moves to its terminal state and cannot be reused.
  </Card>

  <Card title="HTTPS required" icon="lock">
    All API calls and webhooks use HTTPS. Sessions are not transmitted in clear text anywhere.
  </Card>

  <Card title="Concurrent sessions" icon="users">
    Multiple sessions for the same user can be in-flight simultaneously (e.g., a user authenticating from two browser tabs). Each is independent.
  </Card>
</CardGroup>

## Pushing the result to your frontend

Most integrations poll the UIP API from their backend and push the result to their frontend (which is also polling, or listening over SSE/WebSocket). Don't poll UIP directly from a browser using your API key — the key would leak.

<CardGroup cols={3}>
  <Card title="WebSockets" icon="bolt">
    Best for real-time bidirectional updates. Backend polls UIP, pushes status changes over a WS to the frontend.
  </Card>

  <Card title="Server-Sent Events" icon="rss">
    Simpler one-way updates from your server to the browser. Good fit if you only need server → client.
  </Card>

  <Card title="Short polling" icon="rotate">
    Frontend polls your backend every 2–3s. Simple and works everywhere; backend in turn polls UIP.
  </Card>
</CardGroup>
