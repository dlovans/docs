A **primitive** is one verified action against the user's wallet credential. Each
step in a session names a primitive and its `params`. In a wallet-backed completion,
Alentra builds the request, verifies the signed response, and returns a normalized
result.

## identify

Disclose specific government-attested fields, called **blocks**. You request only
what you need; the user consents to exactly that.

```json
{ "primitive": "identify", "params": { "blocks": ["first_name", "last_name", "dob", "country"] } }
```

**Blocks** (mapped to the credential's data elements):

| Block | Meaning |
| --- | --- |
| `first_name` | Given name |
| `last_name` | Family name |
| `dob` | Date of birth (`YYYY-MM-DD`) |
| `gender` | Gender |
| `portrait` | Portrait image |
| `document_number` | Document/license number |
| `issue_date` | Issue date (`YYYY-MM-DD`) |
| `expiry_date` | Expiry date (`YYYY-MM-DD`) |
| `issuing_authority` | Issuing authority |
| `country` | Issuing country selected by the verified credential route |
| `state` | Issuing jurisdiction (e.g. `US-CA`) |
| `address` | Residence address |
| `resident_state` | Residence state |

If `blocks` is omitted, Alentra requests a sensible default (`first_name`,
`last_name`, `dob`, `country`). A requested block the credential doesn't carry
comes back as JSON `null` rather than failing the step.

<Note>
  `country` is the issuing jurisdiction, not residence. A route profile supplies its
  fixed jurisdiction only when the verified credential omits `issuing_country`.
  Google Sandbox uses fictional Utopia data. Claim availability remains profile-
  specific; a requested block not carried by that profile returns `null`.
</Note>

**Result** (`step.completed` → `data.claims`):

```json
{ "claims": { "first_name": "Jane", "last_name": "Doe", "dob": "1990-04-01", "country": "US" } }
```

## age_verify

Prove the holder meets an age threshold **without disclosing their birth date**.
Alentra reads the date from the credential, computes the result, and returns only the
boolean verdict plus the requested threshold. The birth date never reaches your
age-result webhook.

```json
{ "primitive": "age_verify", "params": { "min_age": 21 } }
```

`min_age` is a whole number (1–120; defaults to `18`).

**Result** (`step.completed`):

```json
{ "satisfied": true, "min_age": 21 }
```

<Note>
  This is "anonymous to you, not to Alentra": the signer claims and wallet evidence used
  to compute the result are sealed in the
  [audit record](/core-concepts/verification#audit-ledger), flagged as redacted, and
  never exposed on your webhook or session reads.
</Note>

## sign

The holder's device signs a hash of **one or more documents you provide** — a
what-you-see-is-what-you-sign signature bound to their wallet credential. Because
`sign` carries files, its session is created as `multipart/form-data`.

- Alentra computes the SHA-256 of each file and a single **manifest digest** over all of
  them (in declared order). The wallet signs the manifest; you never supply a hash.
- The user reads the exact bytes Alentra hashed on the hosted page before signing.
- Up to **50 files**, **50 MB** combined. When the step verifies, every referenced
  original is promoted atomically to immutable evidence alongside the result and
  audit. Customers cannot delete those completed originals. An upload that never
  joins a completed sign result remains transient and may be purged after the
  cleanup window.

```bash
curl https://api.alentra.app/v1/sessions \
  -H "Authorization: Bearer $ALENTRA_API_KEY" \
  -F 'steps=[{"primitive":"identify"},{"primitive":"sign","params":{"document_ids":["contract"]}}]' \
  -F 'contract=@contract.pdf'
```

**Result** (`step.completed`; includes the default signer identity fields):

```json
{ "primitive": "sign", "claims": { "first_name": "Jane", "last_name": "Doe", "dob": "1990-04-01", "country": "US" }, "signature": "mdoc:3045...", "document_hash": "sha256:9f86..." }
```

## light_sign

The same device signature, over **short inline terms text** — no file upload. For
consent, acceptance, and disclosure flows.

```json
{ "primitive": "light_sign", "params": { "terms": "I authorize a one-time charge of $49.00." } }
```

- Terms are limited to ~1,000 words (about a 5-minute read).
- Alentra canonicalizes the text (normalizes line endings), hashes it, and the wallet
  signs that hash. The exact terms + hash + signature are kept in the audit, so the
  signature stays provable.

<Warning>
  Terms are shown through the capability URL and retained in cleartext in the
  immutable audit proof. Do not include secrets or unnecessary personal data.
</Warning>

**Result** (`step.completed`; includes the default signer identity fields):

```json
{ "primitive": "light_sign", "claims": { "first_name": "Jane", "last_name": "Doe", "dob": "1990-04-01", "country": "US" }, "signature": "mdoc:3045...", "document_hash": "sha256:1b2c..." }
```

## Chaining steps

Chaining is not a primitive — it's the `steps[]` array. List steps in order and the
user completes them in one session. Alentra creates one logical `step.completed` event
per step and then a terminal `session.*` event; delivery retries can arrive late or
out of order, so use `stage` and the delivery id.

```json
{
  "steps": [
    { "primitive": "identify", "params": { "blocks": ["first_name", "country"] } },
    { "primitive": "age_verify", "params": { "min_age": 18 } },
    { "primitive": "sign", "params": { "document_ids": ["contract"] } }
  ]
}
```

### Gates (pause for a decision)

Set `"gate": true` on a step to **pause** the session after it completes, so your
server can decide whether to continue. Useful when a later step should depend on an
earlier result (e.g. only sign if the identity matched your records).

```json
{ "primitive": "identify", "gate": true }
```

On a gated step the session moves to `awaiting_decision` after `step.completed`.
Resume with [`POST /v1/sessions/{id}/continue`](/api-reference/sessions/continue) or
end it with [`POST /v1/sessions/{id}/stop`](/api-reference/sessions/stop). A gate on
the final step is rejected at create (there's nothing to continue to).
