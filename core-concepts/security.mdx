UIP's security model has two halves: **proving** a result is genuine (covered in
[Verification & trust](/core-concepts/verification)) and **protecting** the data
around it. This page covers the second.

## In transit & at rest

<CardGroup cols={2}>
  <Card title="TLS everywhere" icon="lock">
    All API calls and webhook deliveries are over HTTPS/TLS.
  </Card>
  <Card title="Encrypted responses" icon="envelope">
    The wallet encrypts its response to a **per-request key**; only UIP's server can
    decrypt it. The hosted page relays ciphertext it cannot read.
  </Card>
  <Card title="PII sealed at rest" icon="vault">
    Private session data, sign documents, webhook payloads, signer identity, and
    wallet evidence use application-layer envelope encryption outside local
    development. Ciphertext is bound to its object and purpose with AAD.
  </Card>
  <Card title="Signed webhooks" icon="webhook">
    Every delivery is HMAC-SHA256 signed; verify it before trusting a payload (see
    [Webhooks](/core-concepts/webhooks#verifying-signatures)).
  </Card>
</CardGroup>

## Anti-replay & binding

- **Per-step nonce + origin.** The holder's signature covers a fresh nonce and your
  origin, so a captured response can't be replayed against another request.
- **First-open binding.** The first browser to open a session is bound to it; no
  other device can submit a response for that session.
- **Idempotency.** An `Idempotency-Key` on create makes retried creates safe; webhook
  `X-UIP-Delivery-Id` makes retried deliveries safe to dedupe.

## Data handling & retention

UIP minimizes each disclosure, but it deliberately preserves completed evidence:

- **Only what you request.** `identify` discloses exactly the blocks you ask for.
- **`age_verify` reveals a boolean and threshold** to you, not the birth date; the
  underlying identity is sealed in the audit and flagged redacted.
- **Completed evidence is immutable.** A verified `sign` promotes its original file
  bytes to retained evidence. The completed result, original, audit, signature, and
  integrity links cannot be customer-deleted or time-purged.
- **Transient artifacts are separate.** Only uploads explicitly marked transient
  because no sign result completed may be removed by scheduled cleanup. Terminal
  webhook-delivery copies expire after their operational window; they are not the
  canonical result.
- **No standing identity.** There's no UIP login the user maintains; each session is
  a fresh, consented disclosure from their wallet.

## Your responsibilities

<AccordionGroup>
  <Accordion title="Protect the session URL">
    The hosted `url` is a [capability URL](/core-concepts/sessions#the-hosted-page--capability-urls)
    — deliver it only to the intended user, and don't open it server-side before
    handing it over (that would consume the binding).
  </Accordion>
  <Accordion title="Verify every webhook">
    Recompute the HMAC over the raw body and compare in constant time. Reject stale
    timestamps to bound replay windows. Dedupe on the delivery id.
  </Accordion>
  <Accordion title="Guard your keys">
    Keep your API key and webhook secret server-side. Rotate the webhook secret from
    the dashboard if exposed. API keys are scoped to your business.
  </Accordion>
</AccordionGroup>

## Compliance posture

- Tamper-evident, append-only [audit records](/core-concepts/verification#audit-ledger)
  with sandbox/production UIP signatures and independently checkable RFC 3161 status
  — designed to support evidentiary review, without promising admissibility or legal
  effect.
- Data-minimizing disclosure (selective fields; `age_verify` is anonymous to the
  business, not to UIP).
- Production government-issuer–anchored verification, not document scraping;
  sandbox uses Google's fictional test issuer.

The current managed deployment processes data in US regions, with global Google
Cloud KMS key resources. Production contracts must state the retention period, legal-hold
process, and any required transfer safeguards. Account closure starts a review; it
does not erase completed evidence.
