Webhooks are the primary result-delivery channel: when a step verifies or a session
concludes, UIP `POST`s a signed event to your configured URL. There is no supported
polling loop. If a delivery is missed or dead-lettered, reconcile once with
[`GET /v1/sessions/{id}`](/api-reference/sessions/get).

## Events

| Event | When | Key payload |
| --- | --- | --- |
| `step.completed` | A step verified successfully | `stage`, `primitive`, route provenance for non-anonymous results, and `claims` / `satisfied` / `signature` |
| `session.completed` | All steps done | `session_id`, `status` |
| `session.stopped` | You ended the active session | `session_id`, `status` |
| `session.failed` | A step failed, funds ran out, or an internal error | `session_id`, `reason` |
| `session.expired` | A session/step deadline elapsed | `session_id`, `status` |

Every delivery also echoes your `client_reference_id` and any `metadata`. See
[the event reference](/api-reference/webhooks) for full payloads.

## Delivery guarantees

<CardGroup cols={2}>
  <Card title="Stage-numbered" icon="list-ol">
    Events are created in state order and every step carries `stage` 0, 1, 2… .
    Network retries mean arrival order is not guaranteed; process by stage and make
    terminal handling tolerant of a late step delivery.
  </Card>
  <Card title="At-least-once" icon="rotate">
    Delivery gets up to eight attempts with persisted exponential backoff, then
    becomes `failed`. Each event has a stable `X-UIP-Delivery-Id` — **dedupe on it**
    so a retry is harmless.
  </Card>
  <Card title="Off the request path" icon="bolt">
    Webhooks are queued and sent out-of-band, so a slow endpoint never blocks
    verification — but it does delay your delivery.
  </Card>
  <Card title="Fired after the result is durable" icon="database">
    A step event is sent only after result, audit, applicable usage charge, and any
    document-evidence promotion commit, so a webhook always reflects persisted state.
  </Card>
</CardGroup>

## Verifying signatures

Every delivery is signed with your business's **webhook secret** (`uip_whsec_...`
from the dashboard). Always verify before trusting a payload.

Each request carries:

| Header | Value |
| --- | --- |
| `X-UIP-Event` | The event name |
| `X-UIP-Timestamp` | Unix seconds when signed |
| `X-UIP-Signature` | `HMAC-SHA256(secret, "{timestamp}.{raw_body}")`, hex |
| `X-UIP-Delivery-Id` | Stable per-event id (dedupe key) |

Recompute the HMAC over `timestamp + "." + raw_request_body` and compare in constant
time. **Use the raw bytes** — re-serializing the JSON can change them and break the
match.

<CodeGroup>
```javascript Node (Express)
import crypto from "node:crypto";

// mount with: express.raw({ type: "application/json" })
app.post("/uip-webhook", (req, res) => {
  const ts = req.get("X-UIP-Timestamp");
  const sig = req.get("X-UIP-Signature");
  const expected = crypto
    .createHmac("sha256", process.env.UIP_WEBHOOK_SECRET)
    .update(`${ts}.`)
    .update(req.body) // req.body is the raw Buffer
    .digest();
  const provided = Buffer.from(sig ?? "", "hex");

  const ok =
    provided.length === expected.length &&
    crypto.timingSafeEqual(provided, expected);
  if (!ok) return res.sendStatus(401);

  // Optional: reject stale timestamps to limit replay windows.
  const event = JSON.parse(req.body.toString());
  // ...handle event, dedupe on X-UIP-Delivery-Id...
  res.sendStatus(200);
});
```
```python Python (Flask)
import hmac, hashlib
from flask import request, abort

@app.post("/uip-webhook")
def uip_webhook():
    ts = request.headers.get("X-UIP-Timestamp", "")
    sig = request.headers.get("X-UIP-Signature", "")
    raw = request.get_data()  # raw bytes
    mac = hmac.new(
        SECRET.encode(), f"{ts}.".encode() + raw, hashlib.sha256
    ).hexdigest()
    if not hmac.compare_digest(mac, sig):
        abort(401)
    # ...handle, dedupe on X-UIP-Delivery-Id...
    return "", 200
```
</CodeGroup>

<Tip>
  Respond `2xx` quickly and do heavy work asynchronously. A non-`2xx` (or timeout)
  is treated as a failed delivery and retried.
</Tip>

## Rotating the secret

Generate or rotate your webhook secret from the dashboard. A session snapshots its
webhook URL and secret when the session is created, and its queued deliveries keep
that snapshot. During rotation, accept the old and new secret until in-flight
sessions and retries have drained, then retire the old one.
