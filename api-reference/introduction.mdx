## Welcome to UIP API Reference

This section contains the complete technical documentation for integrating UIP's biometric identity, signature, and messaging services into your application. All endpoints are REST-based with JSON payloads and responses.

<Info>

</Info>

## Authentication

All UIP API endpoints require authentication using Bearer tokens. Include your API key in the Authorization header of every request.

**Header format:**
```
Authorization: Bearer YOUR_API_KEY
```

<Warning>
**Keep your API keys secure.** Never expose them in client-side code, public repositories, or logs. Treat them like passwords.
</Warning>

## Request & Response Format

All requests and responses use JSON format with UTF-8 encoding.

**Required headers:**
- `Authorization: Bearer YOUR_API_KEY`
- `Content-Type: application/json`

## Webhook Integration

Most UIP APIs work asynchronously using webhooks. Instead of polling for results, UIP sends POST requests to your webhook URL when actions complete.

<AccordionGroup>
  <Accordion title="Webhook Security" icon="shield-check">
    Every webhook request includes an `X-UIP-Signature` header containing an HMAC SHA-256 signature. Your webhook secret is a UUID v4 identifier provided when you create your API key.

    **Verify all webhook requests** to ensure they're from UIP and haven't been tampered with.
  </Accordion>

  <Accordion title="Webhook Events" icon="bolt">
    Different APIs send different webhook events:

    - **Identify API:** `authentication_completed`, `authentication_failed`, `session_expired`
    - **Sign API:** `signature_completed`, `signature_declined`, `signature_expired`
    - **Message API:** `message_delivered`, `message_signed`, `message_declined`
  </Accordion>

  <Accordion title="Testing Webhooks" icon="flask">
    Use the [Webhook Test endpoint](/api-reference/endpoint/webhook-test) to verify your webhook handler before going live. It sends a test payload to your URL and reports the response.
  </Accordion>
</AccordionGroup>

## QR Code Flow

The Identify and Sign APIs return QR codes for user authentication. Users scan these codes with the UIP mobile app to complete actions.

**Implementation flow:**

<Steps>
  <Step title="Request Session">
    Call the appropriate endpoint (Identify or Sign) with your webhook URL
  </Step>

  <Step title="Display QR Code">
    Show the returned QR code to users (desktop) or provide app deep link (mobile)
  </Step>

  <Step title="User Scans & Authenticates">
    User opens UIP app, scans code, and authenticates with biometric
  </Step>

  <Step title="Receive Webhook">
    UIP sends result to your webhook URL with verified identity or signature proof
  </Step>
</Steps>

<Info>
**Best Practice:** Always ask users if they're on mobile or desktop first. Mobile users can open the UIP app directly without scanning a QR code.
</Info>

## Error Handling

UIP uses standard HTTP status codes and provides detailed error information in the response body.

**Check HTTP status code first:**
- **2xx:** Success
- **4xx:** Client error (fix your request)
- **5xx:** Server error (retry with exponential backoff)

**Then examine the error code:**
```json
{
  "error": "The requested resource was not found",
  "code": "resource/not-found"
}
```

See the [Error Handling Guide](/core-concepts/error-handling) for complete error code reference.

## Audit Trail

Every UIP action generates a permanent audit record accessible via the Audit API. These records include:

- Complete action details (authentication, signature, message)
- Verified identity information
- Cryptographic proof of action
- Precise timestamps
- Tamper-evident storage

**Audit queries are always free and unlimited.** Store audit IDs from webhook responses to retrieve records anytime.
