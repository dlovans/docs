---
title: 'Create Identify Session'
openapi: 'POST /v1/identify'
---

Create an identity-verification session. Returns a session ID and a QR code your application shows to the user. The user scans it with the UIP app, completes biometric verification, and the result is then available via [`GET /v1/identify/:id`](/api-reference/endpoint/identify-session).

The same endpoint produces four different verification flows depending on the [`mode`](#mode) you pass — share specific identity fields, confirm a user has a valid government ID, check requirements, or check an age threshold without revealing the date of birth.

## Request

### Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Authorization` | Yes | Bearer token with your API key: `Bearer YOUR_API_KEY` |
| `Content-Type` | Yes | `application/json` |

### Body parameters

<ParamField body="mode" type="string" default="profile">
  Which verification flow to run. Determines which other fields are required and what the audit record looks like.

  **Valid values:**
  - `profile` *(default)* — Request specific identity fields (name, date of birth, country, etc.). Produces a `DATA_SHARE` audit.
  - `verify` — Confirm the scanning user has at least one approved, non-expired identity document. Returns a yes/no — no data fields. Produces a `VERIFY` audit.
  - `check` — Confirm the scanning user satisfies a set of requirements (minimum age, allowed nationality, document type, document not expired). Returns pass/fail + which requirements failed. Produces a `CHECK` audit.
  - `age-check` — Confirm the scanning user is at least `min_age` years old. Returns pass/fail only — **never the date of birth**. Produces an `AGE_CHECK` audit.

  All modes are QR-session-based: the user is identified by who scans the QR with the UIP app, not by a value you pass at session creation. Their UIP ID is captured at consume time and returned to you in the polling response.
</ParamField>

<ParamField body="intent" type="string" required>
  Why you're requesting verification. Shown to the user in the UIP app before they accept. Required for `profile` mode; recommended for all modes.

  **Example:** `"Sign in to Example App"` or `"Verify age for content access"`
</ParamField>

<ParamField body="requested_data" type="array">
  Only valid in `profile` mode. Array of identity attributes to request. Omit or pass an empty array to receive all available fields.

  **Valid values** (any subset, max 10):
  - `default_name`, `first_name`, `last_name` — name fields
  - `date_of_birth` — `YYYY-MM-DD`
  - `country`, `address` — location fields
  - `expires_at` — when the user's identity verification expires
  - `personal_number` — national ID number, where available
  - `document_type` — e.g. `passport`, `national_id`
  - `document_number`
  - `gender`

  **Example:** `["first_name", "last_name", "date_of_birth"]`
</ParamField>

<ParamField body="requirements" type="object">
  Only valid in `check` mode. At least one field must be set.

  **Object shape:**
  - `min_age` *(integer, 1–150)* — Minimum age the user must be.
  - `nationalities` *(array of strings)* — Allowed ISO 3166-1 alpha-2 codes. Empty array means any.
  - `document_types` *(array of strings)* — Allowed document types. Empty array means any.
  - `document_valid` *(boolean)* — If `true`, require a non-expired document.

  **Example:** `{ "min_age": 21, "nationalities": ["US", "CA"], "document_valid": true }`
</ParamField>

<ParamField body="min_age" type="integer">
  Only valid (and required) in `age-check` mode. Integer between 1 and 150. The user's actual age is never returned — only whether they meet this threshold.

  **Example:** `18`
</ParamField>

<ParamField body="auditable" type="string" default="full">
  Privacy tier for the audit record this session will produce. **Identify-only** — the Sign API and Message API do not accept this parameter; their audits are always full.

  **Valid values:**
  - `full` *(default)* — Full audit record. Visible in [`GET /v1/audit/:id`](/api-reference/endpoint/audit) with all signer-snapshot data, intent, AdES proof, attachments.
  - `redacted` — A row is written but reads return only the minimal envelope (audit_id, requester_business_name, target_uip_id, action_accepted, created_at). The encrypted content stays on disk and is only surfaced via a legal-case unlock. Use this when the user or your business wants to remain anonymous in routine reads while preserving an evidence record (e.g., adult-industry use cases under 2257 or DSA).
  - `anon` — **No audit row is written at all.** The session still completes; the user still signs; the polling endpoint reports `status: "approved"`. But there is no durable audit record — `GET /v1/audit/:id` returns 404 for the (non-existent) row. Use sparingly.

  Any value other than these three returns `request/invalid-auditable` (400).
</ParamField>

## Response (200 OK)

All modes return the same response shape — a session you display to the user. The actual verification result is then available via [`GET /v1/identify/:id`](/api-reference/endpoint/identify-session) (the response shape there varies by mode).

```json
{
  "session_id": "sess_1a2b3c4d5e6f",
  "expires_at": "2026-05-13T12:35:00Z",
  "qr_code": "<svg xmlns=\"http://www.w3.org/2000/svg\" ...>...</svg>"
}
```

<ResponseField name="session_id" type="string" required>
  Unique identifier for this session. Pass it to `GET /v1/identify/:id` to poll for the result.
</ResponseField>

<ResponseField name="expires_at" type="string" required>
  RFC 3339 timestamp when the session expires. ~5 minutes from creation (controlled by `SESSION_EXPIRY_SECONDS` on the server; default 310 seconds).
</ResponseField>

<ResponseField name="qr_code" type="string" required>
  Raw SVG markup for the QR code to display to the user. **This is NOT a base64 data URI.** Inline it directly in your HTML (`{@html qrCode}` in Svelte, `dangerouslySetInnerHTML` in React) or render it inside an `<svg>` host.

  The encoded payload is `uip|<session_id>|<unix_seconds>` (BankID-style scheme, NOT a URL). On mobile, deep-link to `uip://identify?sid={session_id}&exp={unix_seconds}` to open the UIP app directly.
</ResponseField>

## Examples

### Profile mode — request name and date of birth

<CodeGroup>

```bash cURL
curl -X POST https://api.uip.digital/v1/identify \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "profile",
    "intent": "Sign in to Example App",
    "requested_data": ["first_name", "last_name", "date_of_birth"]
  }'
```

```javascript Node.js
const response = await fetch('https://api.uip.digital/v1/identify', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${API_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    mode: 'profile',
    intent: 'Sign in to Example App',
    requested_data: ['first_name', 'last_name', 'date_of_birth']
  })
});

const { session_id, qr_code, expires_at } = await response.json();
```

```python Python
import requests

response = requests.post(
    'https://api.uip.digital/v1/identify',
    headers={
        'Authorization': f'Bearer {API_KEY}',
        'Content-Type': 'application/json'
    },
    json={
        'mode': 'profile',
        'intent': 'Sign in to Example App',
        'requested_data': ['first_name', 'last_name', 'date_of_birth']
    }
)

session_id = response.json()['session_id']
```

</CodeGroup>

### Age-check mode — privacy-preserving 18+ check

```bash
curl -X POST https://api.uip.digital/v1/identify \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "age-check",
    "intent": "Verify you are 18+ to access this content",
    "min_age": 18,
    "auditable": "redacted"
  }'
```

`auditable: "redacted"` produces a row that proves the verification happened (for 2257 / DSA compliance) without exposing identity details on routine reads.

### Check mode — multiple requirements

```bash
curl -X POST https://api.uip.digital/v1/identify \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "check",
    "intent": "Verify EU national with valid passport",
    "requirements": {
      "min_age": 21,
      "nationalities": ["DE", "FR", "IT", "ES", "NL"],
      "document_types": ["passport"],
      "document_valid": true
    }
  }'
```

### Verify mode — confirm a valid identity document exists

```bash
curl -X POST https://api.uip.digital/v1/identify \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "verify",
    "intent": "Confirm your identity to continue"
  }'
```

## Implementation flow

<Steps>
  <Step title="Create the session">
    `POST /v1/identify` with the appropriate `mode`, `intent`, and any mode-specific fields.
  </Step>

  <Step title="Show the QR code or deep-link">
    Desktop: inline the `qr_code` SVG. Mobile (same-device): button linking to `uip://identify?sid={session_id}&exp={unix_seconds}`.
  </Step>

  <Step title="User scans + signs">
    The user opens the UIP app, sees your intent, accepts, and biometrically signs. The internal API records the audit (or skips it if `auditable: "anon"`).
  </Step>

  <Step title="Poll for the result">
    `GET /v1/identify/:id` every 2–3 seconds until `status === "approved"` or `"expired"`. The result shape varies by mode — see the [Poll Identify Session](/api-reference/endpoint/identify-session) page.
  </Step>

  <Step title="Save the audit_id">
    Persist `audit_id` from the completed response. It's the only handle for retrieving the audit later via [`GET /v1/audit/:id`](/api-reference/endpoint/audit). If you used `auditable: "anon"`, no audit_id is returned.
  </Step>
</Steps>

<Info>
**No webhooks.** Identify uses polling. No redirect URIs, no code exchange.
</Info>

## Errors

See the [Error Handling Guide](/core-concepts/error-handling) for full coverage.

Possible error codes:

- `auth/missing-api-key` — No API key in Authorization header
- `auth/invalid-api-key` — Invalid API key
- `auth/revoked-api-key` — API key has been revoked
- `auth/business-archived` — Business account has been archived
- `auth/verification-required` — Business owner needs to re-run identity verification
- `request/invalid-payload` — Payload missing required fields for the chosen mode, or `requested_data` contains an unknown field, or more than 10 requested fields
- `request/invalid-auditable` — `auditable` was set to a value other than `full`, `redacted`, or `anon`
- `payment/insufficient-funds` — Insufficient account balance
- `internal/server-error` — Failed to create the session
