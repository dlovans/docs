---
title: 'Get Audit Record'
openapi: 'GET /v1/audit/{auditID}'
---

Retrieve the permanent audit record for any completed action. Returns the cryptographic proof (AdES signature, signed payload, public key), the signer's verified identity snapshot, the action details, and signed URLs for any attached documents.

Audit records are append-only, permanently retained, and the canonical evidence layer for legal verification, compliance, and dispute resolution.

## Request

### Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Authorization` | Yes | Bearer token with your API key: `Bearer YOUR_API_KEY` |

### Path Parameters

<ParamField path="auditID" type="string" required>
  The audit ID returned in completed responses from the [Identify](/api-reference/endpoint/identify-session) and [Sign](/api-reference/endpoint/sign-session) GET endpoints, or in the `message` [webhook payload](/api-reference/webhooks).

  **Example:** `"audit_9z8y7x6w5v4u"`
</ParamField>

## Privacy tiers — identify audits only

Audit records produced by the Identify API carry a privacy tier set at session creation via the [`auditable`](/api-reference/endpoint/identify) parameter. The tier controls what fields this endpoint returns:

| Source | `auditable` value | What `GET /v1/audit/:id` returns |
|---|---|---|
| Identify session, default | `full` | The full audit response (below) |
| Identify session, redacted | `redacted` | The minimal envelope (below) |
| Identify session, anonymous | `anon` | **404 — no audit row exists** |
| Sign session | (does not apply) | Full audit response — always |
| Message API | (does not apply) | Full audit response — always |

Only the Identify API supports the privacy-tier system. Sign and message audits cannot be redacted, never produce 404 due to anon, and always return the full response shape below.

## Response shape — full audit

Returned for all sign audits, all message audits, and identify audits where `auditable` is `full` (the default).

<ResponseField name="audit_id" type="string" required>
  Stable identifier for this audit row. Use it to retrieve this same record again.
</ResponseField>

<ResponseField name="audit_type" type="string" required>
  What action produced this record. One of:
  - `DATA_SHARE` — identify session in profile mode (data fields shared)
  - `VERIFY` — identify session in verify mode (identity assertion)
  - `CHECK` — identify session in check mode (requirements check)
  - `AGE_CHECK` — identify session in age-check mode
  - `SIGN` — sign session (lightweight signature)
  - Message-based signatures use the type set on the session at create time

  Use [`requested_fields`](#requested-fields) to distinguish identify subtypes from sign/message.
</ResponseField>

<ResponseField name="requester_business_name" type="string" required>
  Human-readable business name of the requesting business (yours).
</ResponseField>

<ResponseField name="target_uip_id" type="string" required>
  UIP ID of the user who completed the action.
</ResponseField>

<ResponseField name="intent" type="string" required>
  The intent message you provided at session creation. Decrypted before return.
</ResponseField>

<ResponseField name="action_accepted" type="boolean" required>
  Whether the user accepted or declined the action.

  - `true` — User signed / consented / verified
  - `false` — User explicitly declined (currently only possible on message-based signature requests; identify and sign sessions only ever record `true`)
</ResponseField>

<ResponseField name="created_at" type="string" required>
  RFC 3339 / ISO 8601 timestamp of when the user completed the action.
</ResponseField>

<ResponseField name="signer_snapshot" type="object" required>
  Frozen identity snapshot captured at the moment the action was signed. Contains the verified identity fields applicable to this audit type — for `DATA_SHARE` audits, the fields the business [requested](/api-reference/endpoint/identify); for other types, the verifying identity record.

  Embedded structure (subject to change as identity attributes evolve):
  - `target.full_name`, `target.date_of_birth`, `target.nationality`, `target.document_type`, `target.document_number`, `target.issuing_state`, `target.verified_at`, `target.kyc_provider`, `target.kyc_session_id`, `target.liveness_score`, `target.face_match_score`
  - `requester.business_id`, `requester.business_name`
</ResponseField>

<ResponseField name="requested_fields" type="array">
  Array of identity field names the business explicitly requested at session creation. Present only on `DATA_SHARE` (identify-profile) audits; absent from `SIGN`, message-based audits, and other identify modes.
</ResponseField>

<ResponseField name="sign_information" type="string">
  The signed agreement text (for `SIGN` audits) or message body (for message-based audits). Decrypted before return. Empty on identify-only audits.
</ResponseField>

<ResponseField name="attachments" type="array">
  Array of attachment metadata + signed URLs for each PDF attached at signing time. Each entry has `filename` (original filename) and `signed_url` (a 1-hour pre-signed URL pointing at Supabase Storage where the PDF lives). **Download attachments by fetching the `signed_url`**, not by base64-decoding any field in this response.

  Empty array when the audit has no attachments.
</ResponseField>

<ResponseField name="signature" type="string">
  Base64-encoded ECDSA signature over the canonical signed payload. AdES-grade proof. Present on every action that involves a user signing — that's identify-consume, sign, and message-sign.
</ResponseField>

<ResponseField name="signed_payload" type="string">
  The canonical payload the user's device signed. Plain string (JSON-encoded structure) — reconstruct from the audit and verify the signature against this byte sequence.
</ResponseField>

<ResponseField name="signed_payload_hash" type="string">
  SHA-256 hex digest of `signed_payload`. Use it as the document hash if you need to compare signatures across systems.
</ResponseField>

<ResponseField name="public_key" type="string">
  PEM-encoded ECDSA P-256 public key of the signing device. Use it to verify `signature` against `signed_payload`.
</ResponseField>

<ResponseField name="attachment_hashes" type="array">
  SHA-256 hex digests of each attached PDF, in the same order as `attachments[]`. The signature covers these hashes, so any post-signing modification of an attachment invalidates the signature.
</ResponseField>

<ResponseField name="signature_algorithm" type="string">
  Algorithm identifier for `signature`. Currently always `"ES256"` (ECDSA P-256 with SHA-256).
</ResponseField>

## Response shape — redacted audit (identify-only)

Returned only for identify audits where `auditable: "redacted"` was set on the session. Sign and message audits never use this shape. The redacted response returns the minimal envelope — content fields are intentionally omitted until a legal-case unlock is granted by UIP.

```json
{
  "audit_id": "audit_9z8y7x6w5v4u",
  "audit_type": "DATA_SHARE",
  "requester_business_name": "Acme Corp",
  "target_uip_id": "user_def456",
  "action_accepted": true,
  "created_at": "2025-01-11T14:30:00Z",
  "auditable": "redacted"
}
```

<ResponseField name="auditable" type="string">
  Privacy-tier marker. Present and equal to `"redacted"` only on redacted-tier audits; absent from full-tier audits.
</ResponseField>

The `signer_snapshot`, `intent`, `sign_information`, AdES proof fields, and attachments are all absent. The encrypted payload remains in the audit ledger and is recoverable through a legal-case unlock process — contact UIP support if you need a redacted audit unlocked for a legal matter.

## Audits never returned — anon tier (identify-only)

When an identify session is created with `auditable: "anon"`, the consume path writes **no audit row at all**. Querying `GET /v1/audit/:id` for the (non-existent) audit returns `404` (`resource/not-found`). The session still completes — the user still signs, the polling endpoint reports `status: "approved"` — but there is no durable audit. Sign and message audits cannot reach this state.

## Example request

```bash
curl -X GET "https://api.uip.digital/v1/audit/audit_9z8y7x6w5v4u" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## Determining outcome

Use `action_accepted` as the definitive signal of whether the user signed or declined. `audit_type` tells you which API produced the record, not whether the user agreed.

## Errors

See the [Error Handling Guide](/core-concepts/error-handling) for comprehensive error handling strategies.

Possible error codes for this endpoint:

**Authentication Errors:**
- `auth/missing-api-key` — No API key in the Authorization header
- `auth/invalid-api-key` — API key not recognised
- `auth/revoked-api-key` — API key has been revoked

**Request Validation Errors:**
- `request/invalid-payload` — Audit ID missing or not a valid UUID
- `request/not-found` — No such registered route in API service

**Resource Errors:**
- `resource/not-found` — Audit record does not exist OR the audit belongs to a different business OR the row was created at the `anon` privacy tier (no audit produced)

**Data Errors:**
- `data/corrupt` — Audit row is missing required fields (rare; indicates a bug in the writer)
- `data/decryption-failed` — Encrypted intent or sign_information could not be decrypted (signals key drift; surfaces as a 500 to operators)

**Internal Errors:**
- `internal/server-error` — Failed to fetch the audit record or to mint attachment signed URLs

## Implementation flow

<Steps>
  <Step title="Capture the audit_id">
    Identify and Sign: from the completed `GET /v1/identify/:id` or `GET /v1/sign/:id` response. Message: from the `message` webhook payload's `data.audit_id`.
  </Step>

  <Step title="Persist it">
    Store the `audit_id` alongside your internal record of the action. This is the only handle for retrieving the audit later.
  </Step>

  <Step title="Query when needed">
    Call `GET /v1/audit/:id` whenever you need the full record — for compliance reports, dispute investigation, or court submission.
  </Step>

  <Step title="Branch on auditable tier">
    If `auditable === "redacted"`, you have a minimal envelope only. For full content, contact UIP support to initiate a legal-case unlock.
  </Step>

  <Step title="Use signed URLs for attachments">
    `attachments[].signed_url` is a short-lived pre-signed URL (~1 hour). Re-query the audit when you need a fresh URL; do NOT cache the URL itself.
  </Step>
</Steps>

## Best practices

<CardGroup cols={2}>
  <Card title="Save every audit_id" icon="database">
    Persist the audit_id alongside your record of the action. Audits are permanent; you only need to know how to find them.
  </Card>

  <Card title="Treat signed_url as ephemeral" icon="clock">
    Pre-signed URLs expire (~1 hour). Re-query the audit to mint a fresh URL when you need to share the PDF again.
  </Card>

  <Card title="Verify the signature offline" icon="shield-check">
    `signature` + `signed_payload` + `public_key` + `signature_algorithm` together are enough to verify the action cryptographically without trusting UIP's response.
  </Card>

  <Card title="action_accepted, not audit_type" icon="check">
    `audit_type` tells you which API created the record. `action_accepted` tells you what the user did. Branch on the second.
  </Card>
</CardGroup>
