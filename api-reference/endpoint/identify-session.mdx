---
title: 'Poll Identify Session'
openapi: 'GET /v1/identify/{id}'
---

Poll this endpoint every 2–3 seconds after creating a session. When the user completes biometric verification, the response carries a frozen identity snapshot and an `audit_id` you can use to fetch the full audit record.

## Response — pending

```json
{
  "status": "pending",
  "expires_at": "2026-05-13T12:35:00Z"
}
```

The user has not yet scanned the QR or hasn't completed verification. Keep polling.

## Response — approved

```json
{
  "status": "approved",
  "expires_at": "2026-05-13T12:35:00Z",
  "audit_id": "audit_9z8y7x6w5v4u",
  "target_uip_id": "user_def456",
  "signer_snapshot": {
    "target": {
      "uip_id": "user_def456",
      "full_name": "John Doe",
      "date_of_birth": "1990-05-15",
      "nationality": "US",
      "document_type": "passport",
      "document_number": "P1234567",
      "issuing_state": "US",
      "verified_at": "2024-09-01T10:00:00Z",
      "kyc_provider": "didit",
      "kyc_session_id": "didit_abc123",
      "liveness_score": 98.4,
      "face_match_score": 99.1
    },
    "requester": {
      "business_id": "biz_acme",
      "business_name": "Acme Corp"
    }
  }
}
```

The user has scanned the QR, accepted the request, and signed. `audit_id` is the durable handle — store it, and call [`GET /v1/audit/:id`](/api-reference/endpoint/audit) any time you need the full audit (AdES proof, attachments, intent, etc.).

## Response — expired

```json
{
  "status": "expired",
  "expires_at": "2026-05-13T12:35:00Z"
}
```

The session expired before the user completed it. Create a new session if you still need the verification.

## Response — anonymous-tier sessions

If the session was created with `auditable: "anon"`, **no audit row is written at all**. The `approved` response carries only `status` and `expires_at` — no `audit_id`, no `target_uip_id`, no `signer_snapshot`.

```json
{
  "status": "approved",
  "expires_at": "2026-05-13T12:35:00Z"
}
```

You can tell the session was consumed (status flipped from `pending` to `approved`), but you receive no identity data and no evidence record. Use anon only when the business truly does not need to know who scanned (e.g., one-way passwordless flows where the user's session-cookie is enough).

## Fields

<ResponseField name="status" type="string" required>
  One of `"pending"`, `"approved"`, or `"expired"`.

  Note: `"approved"` is the terminal-success state for identify sessions (not `"completed"`). Sign sessions use `"signed"` instead.
</ResponseField>

<ResponseField name="expires_at" type="string" required>
  RFC 3339 timestamp when the session expires (or expired). Mirrors the value returned by the create endpoint.
</ResponseField>

<ResponseField name="audit_id" type="string">
  Permanent audit reference. Present only when `status === "approved"` AND the session's `auditable` tier was `"full"` or `"redacted"` (i.e. an audit row was actually written). Absent on pending, expired, and `auditable: "anon"` responses.
</ResponseField>

<ResponseField name="target_uip_id" type="string">
  UIP ID of the user who completed the verification. Present only on `"approved"` responses.
</ResponseField>

<ResponseField name="signer_snapshot" type="object">
  Frozen identity snapshot captured at the moment the user signed. Same structure as on the audit record. Present only on `"approved"` responses. Use this for the immediate response to your user; use [`GET /v1/audit/:id`](/api-reference/endpoint/audit) when you need the full audit (intent, AdES proof, attachments).
</ResponseField>

## Mode-specific result data

This polling endpoint returns the **same fields regardless of mode** — status + audit envelope. The mode-specific results (`verified`/`active_documents` for `verify`, `passed`/`failed_requirements` for `check`, `passed` for `age-check`) are stored on the session row during consume and accessible from your own polling logic by inspecting the [audit record](/api-reference/endpoint/audit) after `status === "approved"`.

<Info>
**No webhooks needed.** Identify sessions are polled, not pushed.
</Info>

## Errors

- `auth/missing-api-key` — No API key
- `auth/invalid-api-key` — Invalid API key
- `auth/revoked-api-key` — API key has been revoked
- `auth/business-archived` — Business has been archived
- `request/invalid-payload` — Session ID missing in path
- `resource/not-found` — Session does not exist or belongs to a different business
- `internal/server-error` — Failed to fetch the session
