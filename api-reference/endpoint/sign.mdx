---
title: 'Create Sign Session'
openapi: 'POST /v1/sign'
---

Create a signature session. Returns a session ID and a QR code your application shows to the user. The user scans it with the UIP app, sees the agreement, and biometrically signs.

Use this endpoint for lightweight signatures (terms of service, policies, attestations) and for document signatures — attach an optional PDF and the signature covers both the agreement text and the document hash.

Like Identify, Sign is QR-session-based: you do NOT pass a target user upfront. Whoever scans the QR is the signer; their UIP ID is captured at consume time and returned to you when you poll.

## Request

### Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Authorization` | Yes | Bearer token with your API key: `Bearer YOUR_API_KEY` |
| `Content-Type` | Yes | `multipart/form-data` (the endpoint accepts an optional file upload alongside the form fields) |

### Form fields

<ParamField body="intent" type="string" required>
  Why you're requesting the signature. Shown to the user in the UIP app.

  **Example:** `"Sign Terms of Service"` or `"Acknowledge NDA"`
</ParamField>

<ParamField body="signature_information" type="string" required>
  The agreement text the user is signing — the actual content their signature attests to.

  **Example:** `"I agree to the Terms of Service version 2.1 dated May 13, 2026"`
</ParamField>

<ParamField body="mode" type="string" default="document">
  Optional. Mode marker recorded on the resulting audit. Defaults to `"document"`. No other modes are enforced today.
</ParamField>

<ParamField body="document_ref" type="string">
  Optional. An external URL or identifier where the document you're signing lives in your own systems. Stored on the session and included in the audit. Use this when you don't want to upload the PDF to UIP storage but still want a reference in the audit trail.
</ParamField>

<ParamField body="document" type="file">
  Optional. PDF file the user is signing. The server computes a SHA-256 hash of the file at session creation; that hash is included in the signed payload, so post-signing modifications to the PDF invalidate the signature.

  **Constraints:**
  - PDF only (`.pdf` extension required)
  - Max size: 20 MB
  - Single file upload under the form field name `document`

  When present, the file is stored in UIP Storage and a 1-hour pre-signed URL is later returned via the [audit endpoint](/api-reference/endpoint/audit) under `attachments[]`.
</ParamField>

## Response (200 OK)

```json
{
  "session_id": "sess_1a2b3c4d5e6f",
  "expires_at": "2026-05-13T12:35:00Z",
  "qr_code": "<svg xmlns=\"http://www.w3.org/2000/svg\" ...>...</svg>"
}
```

<ResponseField name="session_id" type="string" required>
  Unique session identifier. Pass it to `GET /v1/sign/:id` to poll for the result.
</ResponseField>

<ResponseField name="expires_at" type="string" required>
  RFC 3339 timestamp when the session expires (~5 minutes from creation).
</ResponseField>

<ResponseField name="qr_code" type="string" required>
  Raw SVG markup for the QR code to display. **Not a base64 PNG data URI.** Inline it directly into your HTML.

  The QR encodes `uip|<session_id>|<unix_seconds>` (BankID-style, not a URL). On mobile, deep-link to `uip://sign?sid={session_id}&exp={unix_seconds}` to open the UIP app directly.
</ResponseField>

## Examples

### Lightweight signature — terms of service

```bash
curl -X POST https://api.uip.digital/v1/sign \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -F "intent=Sign Terms of Service" \
  -F "signature_information=I agree to the Terms of Service version 2.1 dated May 13, 2026"
```

### Document signature — sign a PDF

```bash
curl -X POST https://api.uip.digital/v1/sign \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -F "intent=Sign Employment Contract" \
  -F "signature_information=I agree to the terms of the attached employment contract" \
  -F "document=@./contract.pdf"
```

### Document signature with external reference

```bash
curl -X POST https://api.uip.digital/v1/sign \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -F "intent=Sign Lease Agreement" \
  -F "signature_information=I agree to the lease terms at https://docs.example.com/lease-2026.pdf" \
  -F "document_ref=https://docs.example.com/lease-2026.pdf"
```

## Implementation flow

<Steps>
  <Step title="Create the session">
    `POST /v1/sign` with `intent` + `signature_information`. Optionally attach a PDF or a `document_ref`.
  </Step>

  <Step title="Show the QR or deep-link">
    Desktop: inline the `qr_code` SVG. Mobile (same-device): button linking to `uip://sign?sid={session_id}&exp={unix_seconds}`.
  </Step>

  <Step title="User scans and signs">
    The user opens the UIP app, sees your intent + signature_information (and the document filename if you attached one), and biometrically signs.
  </Step>

  <Step title="Poll for the result">
    `GET /v1/sign/:id` every 2–3 seconds until `status === "signed"` or `"expired"`. See [Poll Sign Session](/api-reference/endpoint/sign-session) for the response shape.
  </Step>

  <Step title="Save the audit_id">
    Persist `audit_id` from the signed response. The full audit (AdES proof, attachments via signed URL, signed payload hash) is available via [`GET /v1/audit/:id`](/api-reference/endpoint/audit).
  </Step>
</Steps>

<Info>
**No webhooks.** Sign uses polling.
</Info>

## Errors

See the [Error Handling Guide](/core-concepts/error-handling) for full coverage.

Possible error codes:

- `auth/missing-api-key` — No API key in Authorization header
- `auth/invalid-api-key` — Invalid API key
- `auth/revoked-api-key` — API key has been revoked
- `auth/business-archived` — Business has been archived
- `auth/verification-required` — Business owner needs to re-run identity verification
- `request/invalid-payload` — Failed to parse multipart form, or `intent` / `signature_information` empty
- `request/invalid-file-type` — Uploaded file is not a PDF
- `request/file-too-large` — PDF exceeds 20 MB
- `payment/insufficient-funds` — Insufficient account balance
- `internal/server-error` — Failed to create the session
