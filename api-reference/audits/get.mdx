---
title: "Get an audit record"
description: "GET /v1/audits/{id}"
---

<Note>
  `GET https://api.alentra.app/v1/audits/{id}` · Auth: `Bearer` API key
</Note>

Returns the owning business's audit disclosure for one verified step. It includes
the canonical record hash, per-session chain link, Alentra public signature, RFC 3161
status/token, primitive proof, and the encrypted blocks required to reproduce the
hash. For `identify`, `sign`, and `light_sign`, it also decrypts `signer` and
the available `wallet_evidence` so you can re-run wallet verification. A simulated
development result has no wallet evidence. `age_verify` deliberately withholds both
and returns only its boolean/threshold proof.

The audit id is `{session_id}_{stage}`. You receive it on the
[`step.completed`](/api-reference/webhooks#step-completed) webhook and on every
result in [`GET /v1/sessions/{id}`](/api-reference/sessions/get).

## Path parameters

<ParamField path="id" type="string" required>
  The audit id, e.g. `sess_abc123_0`.
</ParamField>

## Response

<ResponseField name="id" type="string">The audit id.</ResponseField>
<ResponseField name="session_id" type="string">The source session.</ResponseField>
<ResponseField name="business_id" type="string">The owning business; part of the canonical hash/AAD context.</ResponseField>
<ResponseField name="environment" type="string">Signed trust/data plane.</ResponseField>
<ResponseField name="credential_profile" type="string">Signed credential trust profile id.</ResponseField>
<ResponseField name="jurisdiction" type="string">Signed route jurisdiction; withheld from the business tier for `age_verify`.</ResponseField>
<ResponseField name="stage" type="integer">Zero-based step position.</ResponseField>
<ResponseField name="primitive" type="string">`identify` · `age_verify` · `sign` · `light_sign`.</ResponseField>
<ResponseField name="wallet" type="string">The wallet adapter that fulfilled the step.</ResponseField>
<ResponseField name="delivery_mode" type="string">The response transport, currently `DC_API`.</ResponseField>
<ResponseField name="proof" type="object">
  Cleartext, PII-minimized proof fields plus `evidence_digest` when wallet evidence
  was captured.
  <Expandable title="proof by primitive">
    <ResponseField name="claims_digest" type="string">`identify` — RFC 8785/SHA-256 commitment to the disclosed claims.</ResponseField>
    <ResponseField name="satisfied" type="boolean">`age_verify` — threshold result.</ResponseField>
    <ResponseField name="min_age" type="integer">`age_verify` — requested threshold.</ResponseField>
    <ResponseField name="document_hash" type="string">`sign`/`light_sign` — signed manifest/terms digest.</ResponseField>
    <ResponseField name="signature" type="string">`sign`/`light_sign` — compact DeviceAuth signature value. Verify it with `wallet_evidence`; it is not a standalone proof.</ResponseField>
    <ResponseField name="terms" type="string">`light_sign` — exact canonical terms.</ResponseField>
    <ResponseField name="documents" type="object[]">`sign` — ordered per-file `{ id, hash }` manifest.</ResponseField>
    <ResponseField name="evidence_digest" type="string">Commitment to the complete wallet evidence object.</ResponseField>
  </Expandable>
</ResponseField>
<ResponseField name="signer" type="object">Decrypted signer claims for non-age primitives only.</ResponseField>
<ResponseField name="wallet_evidence" type="object">For wallet-backed non-age primitives, decrypted ISO mdoc verification material: exact DeviceResponse, transcript, nonce/content binding, origin, profile, and verification time.</ResponseField>
<ResponseField name="signer_sealed" type="object">Encrypted signer block retained so an offline tool can reproduce `record_hash` without plaintext.</ResponseField>
<ResponseField name="wallet_evidence_sealed" type="object">Encrypted wallet-evidence block retained for canonical hash reproduction.</ResponseField>
<ResponseField name="prev_hash" type="string">Previous stage's `record_hash`, or `0x000` at stage 0.</ResponseField>
<ResponseField name="record_hash" type="string">SHA-256 over the RFC 8785 canonical record content.</ResponseField>
<ResponseField name="alentra_signature" type="object">Sandbox/production only. Alentra's P-256/SHA-256 signature over the record digest: `{ version, alg: "ES256", kid, issuer, signed, value }`. `value` is base64url ASN.1 DER ECDSA, not a compact JWS.</ResponseField>
<ResponseField name="ts_status" type="string">Present when trusted timestamping is configured. Immutable synchronous outcome: `ok` when the initial RFC 3161 request returned a validated token; `deferred` when it failed. A recovered record keeps `deferred` and adds `ts_backfilled`.</ResponseField>
<ResponseField name="ts_token" type="string">Base64 RFC 3161 token captured initially or merged after recovery.</ResponseField>
<ResponseField name="ts_backfilled" type="boolean">`true` when a deferred token was recovered in the append-only timestamp collection.</ResponseField>
<ResponseField name="occurred_at" type="string">ISO 8601 verification time.</ResponseField>
<ResponseField name="client_reference_id" type="string">Your non-PII correlation key, if supplied; part of the immutable record.</ResponseField>

## Offline verification

Download the public keys from
`https://api.alentra.app/.well-known/alentra-audit-jwks.json`, save this response, and use
`cmd/verify-audit` from the `alentra-services` source tree. It reproduces the record
hash and verifies the required sandbox/production Alentra signature; when
`wallet_evidence` is present, it also re-runs the ISO mdoc proof against roots you
supply. See
[Verification & trust](/core-concepts/verification#verify-a-record-offline).

<Warning>
  Integrity verification does not by itself establish admissibility, legal effect,
  or suitability for your transaction. Preserve the complete response and original
  signed files, and obtain legal advice where needed.
</Warning>

<Note>
  Re-running the withheld `age_verify` wallet proof requires an approved legal-tier
  disclosure. It is not available through a customer API key.
</Note>
