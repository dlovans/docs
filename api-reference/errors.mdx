title: "Errors"
description: "Error format and HTTP status codes."
---

Errors use standard HTTP status codes and a JSON body with a human-readable message:

```json
{ "error": "body must include a non-empty steps[]" }
```

## Status codes

| Status | Meaning | Typical cause |
| --- | --- | --- |
| `400` | Bad Request | Malformed body, unknown primitive, invalid `min_age`, bad `steps[]` |
| `401` | Unauthorized | Missing/invalid API key, or its business is archived/billing-frozen |
| `402` | Payment Required | Insufficient balance to start the session |
| `403` | Forbidden | A known session belongs to another active business (or an archive race was detected) |
| `404` | Not Found | Unknown session or audit id |
| `409` | Conflict | Wrong session state or concurrent modification |
| `422` | Unprocessable | No webhook configured for your business (set one in the dashboard) |
| `5xx` | Server / Unavailable | Transient error (`500`) or a dependency not configured (`503`) |

## Notes

<AccordionGroup>
  <Accordion title="402 — fund your balance">
    Creating a session checks eligibility against your balance (and overdraft floor /
    credit limit). Top up in the dashboard, then retry. See [Pricing](/pricing).
  </Accordion>
  <Accordion title="422 — webhook required">
    Webhooks are the required primary result channel, so a session can't be created
    until your business has a webhook URL. Configure it in the dashboard; use GET
    only to reconcile a missed event.
  </Accordion>
  <Accordion title="409 — state">
    Continue returns a conflict when the session is not waiting at a gate. Concurrent
    mutations can also conflict and should be retried. Stop is idempotent for an
    existing terminal session. A reused create `Idempotency-Key` returns its original
    session.
  </Accordion>
  <Accordion title="Retrying safely">
    Retry `5xx` and network errors with backoff. Make `POST /v1/sessions` retries safe
    with an `Idempotency-Key`.
  </Accordion>
</AccordionGroup>
