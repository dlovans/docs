This walks you from zero to a verified webhook in four steps. The example runs a
**business-redacted age check** (`age_verify`), but the flow is identical for any primitive.

<Steps>
  <Step title="Get your API key + set a webhook">
    Sign up at [uip.digital](https://uip.digital), create a business, and from the
    dashboard copy your **API key** (`uip_...`) and set your **webhook URL**.

    <Warning>
      A webhook URL is **required**. Webhooks are the primary result channel; the
      session GET exists only for one-off reconciliation, not polling. A session won't
      be created until your business has a webhook configured. You'll also need your **webhook signing secret**
      (`uip_whsec_...`) from the dashboard to verify deliveries (step 4).
    </Warning>
  </Step>

  <Step title="Create a session">
    Server-to-server, with your API key. List the `steps` you want verified.

    <CodeGroup>
    ```bash cURL
    curl https://api.uip.digital/v1/sessions \
      -H "Authorization: Bearer $UIP_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "steps": [ { "primitive": "age_verify", "params": { "min_age": 21 } } ],
        "client_reference_id": "order_4815"
      }'
    ```
    ```javascript Node
    const res = await fetch("https://api.uip.digital/v1/sessions", {
      method: "POST",
      headers: {
        Authorization: `Bearer ${process.env.UIP_API_KEY}`,
        "Content-Type": "application/json"
      },
      body: JSON.stringify({
        steps: [{ primitive: "age_verify", params: { min_age: 21 } }],
        client_reference_id: "order_4815"
      })
    });
    const session = await res.json();
    ```
    </CodeGroup>

    UIP returns the session and its hosted URL:

    ```json
    {
      "id": "sess_abc123",
      "status": "created",
      "url": "https://uip.digital/s/sess_abc123",
      "expires": "2026-06-07T12:34:56Z",
      "client_reference_id": "order_4815"
    }
    ```
  </Step>

  <Step title="Send the user to the hosted page">
    Hand the user `session.url`. UIP's hosted page invokes the W3C Digital
    Credentials API; the browser owns the same-device or cross-device handoff. Do
    not call UIP's hosted-page transport endpoints or build a polling/QR protocol of
    your own.
  </Step>

  <Step title="Receive the signed result">
    When the wallet responds and UIP verifies it, you get a `POST` to your webhook
    URL — one logical `step.completed` per step plus a terminal `session.*` event.
    Network retries can change arrival order.

    ```http
    POST /your-webhook
    X-UIP-Event: step.completed
    X-UIP-Timestamp: 1749300000
    X-UIP-Signature: 9f86d081884c7d65...
    X-UIP-Delivery-Id: sess_abc123:step.completed:0

    {
      "event": "step.completed",
      "delivery_id": "sess_abc123:step.completed:0",
      "sent": "2026-07-13T12:01:30Z",
      "data": {
        "session_id": "sess_abc123",
        "client_reference_id": "order_4815",
        "stage": 0,
        "primitive": "age_verify",
        "audit_id": "sess_abc123_0",
        "satisfied": true,
        "min_age": 21
      }
    }
    ```

    Always [verify the signature](/core-concepts/webhooks#verifying-signatures)
    before trusting a delivery:

    <CodeGroup>
    ```javascript Node
    import crypto from "node:crypto";

    function verify(req, secret) {
      const ts = req.headers["x-uip-timestamp"];
      const sig = req.headers["x-uip-signature"];
      const expected = crypto
        .createHmac("sha256", secret)
        .update(`${ts}.`)
        .update(req.rawBody) // raw bytes, not re-serialized JSON
        .digest();
      const provided = Buffer.from(sig ?? "", "hex");
      return provided.length === expected.length &&
        crypto.timingSafeEqual(provided, expected);
    }
    ```
    ```python Python
    import hmac, hashlib

    def verify(timestamp: str, raw_body: bytes, signature: str, secret: str) -> bool:
        mac = hmac.new(secret.encode(), f"{timestamp}.".encode() + raw_body, hashlib.sha256)
        return hmac.compare_digest(mac.hexdigest(), signature)
    ```
    </CodeGroup>
  </Step>
</Steps>

## Compose multiple steps

Real flows often need more than one verified action — for example, *identify the
signer, then sign the contract*. List them in order; the user completes them in one
session and UIP creates one logical webhook per step. Use `stage` and the stable
delivery id because retries can arrive late or out of order.

```json
{
  "steps": [
    { "primitive": "identify", "params": { "blocks": ["first_name", "last_name", "country"] } },
    { "primitive": "sign", "params": { "document_ids": ["contract"] } }
  ]
}
```

<Note>
  `sign` carries documents, so its session is created as `multipart/form-data` — see
  [Primitives → sign](/core-concepts/primitives#sign) and
  [Create a session](/api-reference/sessions/create).
</Note>

## Choose a test mode

- **Local development:** run the service with `ENVIRONMENT` unset and the web app in
  its development build. The hosted page then exposes **Simulate**, and backend
  issuer trust is relaxed. These results are for development only.
- **Google Sandbox:** `ENVIRONMENT=SANDBOX` uses Google's published request signer,
  RP metadata, Utopia ID Pass profile, and sandbox IACA. It runs the real encrypted
  wallet path with strict sandbox issuer trust; install a Utopia test ID in Google
  Wallet first.
- **Production:** `ENVIRONMENT=PRODUCTION` accepts only ledger routes explicitly
  promoted to live, with provider credentials and authenticated issuer roots for each
  exact credential profile.

The API and webhook contract are identical in all three modes. See
[Wallets & environments](/core-concepts/wallets#environments).
